Part 2 · 1

첫 프롬프트가 품질을 크게 좌우합니다

대충 말하고 나중에 요구사항을 흘리면 Claude도 헤맵니다.

대화는 길어지고, 처음 잘못 잡은 방향은 context에 계속 남습니다.

좋은 첫 프롬프트의 5가지

Goal

무엇을 바꾸거나 만들 것인가

Context

관련 파일, 로그, 에러, 예시

Constraints

아키텍처, 보안, 스타일 제약

Done when

완료 기준, 테스트, 기대 출력

Workflow

탐색 · 계획 · 구현 · 리뷰 순서

Part 2 · 1

Bad vs Good Prompt

Bad
결제 버그 고쳐줘.
Good
Goal: 만료 카드 사용자 checkout flow 실패 버그 수정 Context: src/payments/, src/auth/session.ts session timeout 후 재시도 시 401 발생 [로그 첨부] Constraints: - 기존 payment middleware 패턴 준수 - 새 dependency 추가 금지 - token refresh 로직 우회 금지 Done when: 1. failing test 먼저 추가 2. 구현 후 관련 테스트 실행 3. root cause 요약 Workflow: 계획 승인 전에는 파일 수정 금지

절대 모호한 프롬프트로 AI가 추측하게 만들지 마세요.

Part 2 · 7

언제 새 세션을 시작하나

  • 새 기능은 새 세션
  • 새 버그는 새 세션
  • 새 PR은 새 세션
  • 관련 없는 질문은 새 세션이나 /btw
  • 두 번 이상 빗나간 작업은 /clear 후 더 좋은 첫 프롬프트

같은 기능의 문서 작성처럼 직전 맥락이 유용한 작업은 이어가도 됩니다.

하지만 인증 디버깅 중에 갑자기 다른 API endpoint를 만들기 시작하면 /clear가 낫습니다.

→ 디버깅 로그와 실패 접근법이 새 작업에는 노이즈입니다.

Rewind는 Correction보다 강력하다

Correction
그거 아니야. 다른 방식으로 해.

잘못된 접근과 그 출력이 context에 남습니다.
Claude가 계속 의식하게 됩니다.

Rewind
/rewind 방금 접근 A는 쓰지 마라. foo module은 해당 API를 expose하지 않는다. 아까 읽은 파일 정보는 유지하되, 바로 접근 B로 가라.

잘못된 가지를 잘라내고 다시 시작하는 게 낫습니다.

Compact vs Clear

/compact

대화를 모델이 요약해 이어감 — 편하지만 lossy

좋은 경우

긴 작업의 핵심 흐름은 유지
중간 조사 결과가 많지만 요약으로 충분
중요 정보를 Claude가 비교적 잘 안다

/clear

빈 context로 새롭게 시작

좋은 경우

작업이 바뀐다
실패한 접근이 context를 오염시켰다
다음 방향을 예측하기 어렵다

Part 2 · 9

좋은 Clear Handoff

새 세션용 요약: 우리는 auth middleware를 refactor 중이다. 중요 제약: - public route는 기존 allowlist를 유지해야 한다 - token refresh는 src/auth/refresh.ts의 기존 함수 사용 - approach A는 실패 — foo module이 필요한 API를 expose하지 않는다 관련 파일: - src/auth/middleware.ts - src/auth/refresh.ts - tests/auth/middleware.test.ts 다음 작업: 1. session timeout 후 refresh가 호출되는 failing test 작성 2. middleware 수정 3. auth test만 실행
Part 2 · 10

Bad autocompact를 피하라

자동 compact는 context가 거의 찼을 때 발생합니다 — 이미 context rot이 심한 상태일 수 있습니다.

→ 긴 작업에서는 proactive compact가 좋습니다.

/compact focus on the auth refactor. Preserve: - modified files - failed approaches - exact test commands - remaining TODO Drop: - unrelated lint debugging - exploratory logs

compact도 그냥 맡기는 게 아니라 지시해야 합니다.

PART 03

Prompt Caching

왜 모델·도구를 함부로 바꾸면 안 되는가

Part 3 · 1

Prompt caching은 prefix match다

요청의 앞부분이 동일하면 캐시를 재사용할 수 있습니다.

API 계층: tools → system → messages

① tools
② system
③ messages

TTL과 비용

기본 TTL은 5분, 1시간 TTL은 추가 비용.
cache read 0.1x · 5분 write 1.25x · 1시간 write 2x.

CLAUDE.md는 system prompt가 아니라 messages 쪽 context로 주입됩니다.

Stable context는 앞에. Dynamic updates는 뒤에.
Part 3 · 2

"같은 말을 반복하라"는 뜻이 아닙니다

이런 생각이 들 수 있습니다 —

"매번 첫 문장을 똑같이 써야 하나?"
"prefix가 내 채팅 입력의 서두를 말하는 건가?"

결론: 아닙니다.

prefix match는 우리가 채팅창에 입력하는 문장 첫머리만을 말하는 게 아닙니다.

Claude Code가 모델에게 보내는
전체 context의 안정적인 앞부분을 깨뜨리지 말자
Part 3 · 3

실제로 모델에게 보내지는 것

우리는 "이 버그 고쳐줘" 한 줄을 입력했지만 —

[사용 가능한 tool 정의들] [Claude Code의 고정 system prompt] [messages: 프로젝트의 CLAUDE.md와 rules context] [messages: 현재 세션 context] [messages: 지금까지의 대화 기록] [messages: Claude가 읽은 파일 내용] [messages: 도구 실행 결과] [messages: 마지막 사용자 메시지 "이 버그 고쳐줘"]

우리가 입력한 프롬프트는 전체 요청 중 맨 끝에 붙는 작은 부분일 수 있습니다.

앞부분이 이전 요청과 같으면 처리 결과를 재사용 → 캐시 적중.

Part 3 · 4

static content first, dynamic content last

자주 바뀌는 정보를 tools, system, messages 초반에 두면? — prefix가 흔들립니다.

Bad — system prompt에 시간 박기
현재 시간은 2026-05-09 10:01:03입니다.

↓ 다음 요청

현재 시간은 2026-05-09 10:01:17입니다.

→ 초반 prefix가 바뀌어 뒤의 긴 context cache hit가 어려워집니다

Good

동적 정보는 뒤쪽 메시지나 tool result로 전달

  • 시간
  • 현재 파일 상태
  • 방금 실패한 테스트 로그
Part 3 · 5

그러면 프롬프트 앞에
항상 같은 말을 써야 하나?

예를 들어 매번 이렇게 시작할 필요가 있을까요?

너는 뛰어난 시니어 개발자야. 다음 작업을 단계적으로 수행해줘.

아닙니다.

사용자 메시지는 보통 전체 요청의 뒤쪽에 붙기 때문입니다.

진짜 중요한 것은

Claude Code system prompt tool 정의 CLAUDE.md session context 도구 구성 모델·가격 가정 긴 대화의 앞부분

→ 안정적인 정보와 동적인 정보를 잘 분리하는 것

Part 3 · 6

CLAUDE.md에는 안정적인 정보만

✓ 넣기 좋은 것 (안정)

  • 테스트 명령
  • 빌드 명령
  • lint 명령
  • typecheck 명령
  • 팀 코딩 규칙
  • 아키텍처 원칙
  • PR 규칙
  • 자주 하는 실수 / 금지된 패턴

✗ 넣지 말 것 (자주 바뀜)

  • 오늘 작업 상태
  • 현재 실패 중인 테스트 로그
  • 방금 시도했다가 실패한 접근
  • 임시 TODO
  • 현재 ticket의 최신 코멘트
  • 오늘 날짜
Part 3 · 7

모델을 중간에 바꾸면 가정이 달라질 수 있다

cache hit는 동일한 prefix를 전제로 하고, 가격과 지원 조건은 모델별로 다릅니다.

직관

"이번 질문은 쉬우니까 Haiku로 바꾸면 더 싸지 않을까?"

안전한 설명

모델 전환이 항상 비용 절감으로 이어진다고 단정하지 말 것.
긴 context 전체보다 handoff summary를 넘기는 편이 안전합니다.

Part 3 · 8

Tool set도 중간에 바꾸지 마라

tool 정의는 prompt prefix의 일부 — 변경하면 tools / system / messages cache 전체가 무효화될 수 있습니다.

위험한 직관들

  • "plan mode니까 read-only tool만 남기면 되지 않을까?"
  • "GitHub 작업 아니니까 GitHub MCP는 빼면 되지 않을까?"
  • "필요할 때만 추가하면 context도 줄고 좋은 거 아닌가?"

→ stable stub + tool search / defer_loading이 더 cache-safe합니다

Part 3 · 9

사용자가 기억해야 할 4가지

01

매번 같은 프롬프트 서두를 쓸 필요 없다

"너는 뛰어난 시니어 개발자야..." — 이건 핵심이 아닙니다

02

안정 정보와 동적 정보를 분리하라

안정 → CLAUDE.md / 동적 → 메시지·task context

03

긴 세션 중간에 모델·tool 구성을 함부로 바꾸지 마라

모델은 비용/latency 가정, tools는 cache prefix를 흔듭니다

04

같은 작업은 같은 세션을 유지하라

새 작업·새 버그·새 PR이면 새 세션

PART 04

MCP, Skills, Hooks, Subagents

Claude Code를 확장하는 실전 도구들

Part 4 · 1

Claude Code 확장의 네 가지 축

MCP

외부 도구와 데이터에 접근

Skill

팀의 업무 절차와 판단 기준을 캡슐화

Subagent

별도 context에서 조사·검증·전문 역할 수행

Hook

이벤트와 matcher가 맞으면 자동 실행되는 automation

+ slash command, plugin, CLI script, Makefile, log file, browser automation

Part 4 · 2

MCP - 외부 시스템에 접근하는 능력

Notion · Linear · GitHub · Figma · DB · monitoring · docs

현실적인 우선순위

① Makefile · shell script · CLI · log file로 해결
② 필요한 경우 MCP를 붙인다
③ MCP + Skill 결합으로 팀 workflow 표준화

MCP도 비용과 실패 가능성이 있습니다. 다만 tool search / defer_loading으로 완화됩니다.

Part 4 · 3

MCP만으로는 부족하다

MCP는 도구 접속 능력 — 어떤 순서로, 어떤 기준으로, 어떤 예외 처리와 함께 쓸지는 알려주지 않습니다.

MCP는 Notion · Linear · GitHub · Figma에 접속하는 능력
Skill은 그 도구들을 우리 팀 방식대로 쓰는 레시피

→ 절차를 Skill로 캡슐화해야 팀 workflow가 됩니다

Part 4 · 4

Skills - 긴 프롬프트가 아니라 업무 절차

"무엇을 할 수 있는지"가 아니라, "우리 팀은 이 일을 어떻게 하는지"를 가르치는 방법.

Claude Code에서는 주로 .claude/skills/ 아래 Custom Skills를 만들어 팀 workflow를 정의합니다.

좋은 Skill 후보

스프린트 계획

Figma → 개발 handoff

GitHub issue 수정

Sentry 기반 PR 분석

릴리즈 체크리스트

고객 온보딩 워크스페이스

표준 PR 리뷰

로그 triage

migration / incident 관리

Part 4 · 5

Progressive Disclosure

Skill은 보통 세 단계로 로드됩니다.

항상 로드

YAML frontmatter

name, description

관련 시 로드

SKILL.md 본문

핵심 절차 · 판단 기준

필요할 때만

references / scripts / assets

긴 예시 · 템플릿

.claude/skills/ └── fix-issue/ ├── SKILL.md ├── references/ │ ├── issue-triage.md │ └── pr-review-checklist.md └── scripts/ └── validate-fix.sh
Part 4 · 6

Description이 Skill의 성패를 좌우한다

Claude는 Skill 본문을 항상 읽지 않습니다. 먼저 YAML description을 보고 사용 여부를 판단합니다.

Bad
name: project-helper description: Helps with the project.

너무 넓고 모호함

Good
name: fix-github-issue description: GitHub issue를 분석하고 수정합니다. 사용자가 "fix issue", "GitHub issue", "issue #123", "bug ticket" 등을 언급할 때 사용합니다.

Skill이 잘 안 불린다면? → 모델 탓 말고 description부터 보세요

Part 4 · 7

좋은 Skill은
구체적 유스케이스에서 시작한다

Bad — 너무 넓고 모호함
name: project-helper description: Helps with the project.
Good — 명확한 trigger
name: fix-github-issue description: GitHub issue를 분석하고 수정합니다. "fix issue", "GitHub issue", "issue #123", "bug ticket" 등 언급 시 사용.

Skill을 만들기 전에 정의할 것

  • 사용자가 무엇을 달성하려는가? 필요한 입력은?
  • 필요한 단계는? 어떤 도구를 써야 하는가?
  • 어떤 도메인 지식이 필요한가? 완료 기준은?
  • 실패 시 어떻게 복구하는가?

Skill은 "멋진 프롬프트 모음"이 아닙니다 — 반복 가능한 업무 프로세스입니다.

Part 4 · 8

Skill은 반드시 테스트해야 한다

Skill도 코드처럼 — 만들어놓고 끝이 아닙니다. 팀 workflow처럼 계속 개선해야 합니다.

Trigger Test

  • 명확한 요청에서 켜지는가
  • 다른 표현에서도 켜지는가
  • 관련 없는 요청에서 꺼지는가

Functional Test

  • 원하는 산출물이 나오는가
  • 도구 호출이 성공하는가
  • 예외 처리가 되는가
  • 검증 gate가 실행되는가

Baseline 비교

  • 메시지 수가 줄었는가
  • 토큰이 줄었는가
  • 실패 호출이 줄었는가
  • 결과 품질이 일관적인가
Part 4 · 9

Hooks - 말로 부탁하지 말고 시스템으로 강제하라

Hooks는 특정 lifecycle 지점에서 shell command, HTTP endpoint, LLM prompt를 실행합니다.

CLAUDE.md의 지시

advisory — Claude가 잊을 수 있다

Hook

matcher가 맞으면 자동 실행된다

말로 부탁해도 되는 것 → CLAUDE.md / Skill
정해진 lifecycle에서 자동화할 것 → Hook
Part 4 · 10

Hook에 적합한 작업

Hook은 deterministic한 규칙에 적합합니다.

✓ Hook에 넣기 좋은 작업

  • 파일 수정 후 formatter / linter 실행
  • migration folder write 차단
  • production config 수정 차단
  • rm -rf, sudo, force push 차단
  • test, build, verifier 실행
  • session start 시 git status 주입
  • prompt submit 시 sprint context 주입
  • compact 전 transcript 백업

✗ Hook에 부적합한 작업

  • 모호한 설계 선호
  • 상황별 판단이 필요한 아키텍처 선택
  • 긴 문서 설명
  • 긴 로그 전체 무조건 주입
  • 민감 파일을 읽는 스크립트
Part 4 · 11

자주 쓰는 Hook 이벤트

아래는 대표 이벤트입니다. 최신 reference에는 Setup, PostToolBatch, FileChanged, SessionEnd 등 더 많은 이벤트가 있습니다.

PreToolUse
tool 실행 전 — 위험 명령 차단, file path 검증
PermissionRequest
permission dialog 직전 — npm test 자동 승인 등
PostToolUse
tool 종료 후 — Prettier, formatter 실행
PreCompact
compaction 전 — transcript 백업, 결정 추출
SessionStart
세션 시작/resume — git status, TODO 주입
Stop
응답 종료 직전 — test pass 전 stop 차단
SubagentStop
subagent 종료 — output 품질 검증
UserPromptSubmit
prompt 제출 직후 — sprint context 주입

⚠ Command hook은 사용자 권한으로 shell command를 실행합니다. 보안 review 필수.

Part 4 · 12

Hook 보안

Command hook은 사용자 권한으로 shell command를 실행합니다. 강력한 만큼 사고가 나면 크게 납니다.

필수 원칙

  • stdin input validation
  • shell variable quoting
  • absolute path 사용
  • 민감 파일 처리 금지
  • .env, credential 접근 금지
  • hook 설정 변경은 review
  • production command 차단
"편하게 하려고 만든 자동화"가 production config를 건드리거나 secret을 로그에 남기면 안 됩니다.
Part 4 · 13

Subagents - root context를 지키는
가장 강력한 도구

상위 agent가 별도 context window를 가진 agent를 호출 → 핵심 가치는 context 보호.

main context를 오염시키는 작업들

긴 조사 대량 파일 읽기 verbose test output 보안 리뷰 성능 리뷰 문서 조사
이 tool output을 나중에 다시 봐야 하는가,
아니면 결론만 필요한가?

결론만 필요하다면 subagent가 좋습니다.

Part 4 · 14

Explore Subagent

새 repo에서 작업을 시작할 때 — Claude는 관련 파일을 찾기 위해 많은 파일을 읽습니다.
이걸 main context에서 다 하면 context가 빨리 더러워집니다.

Find the code that implements the chapter diff view. Look for: - templates that render diffs - Python code using difflib - JavaScript for diff rendering - CSS styles for red/green backgrounds Search templates/, static/, blog/. Return relevant files, line numbers, and implementation summary. Do not modify files.

→ parent context에는 전체 파일이 아니라 탐색 요약만 들어옵니다.

Part 4 · 15

Parallel Subagents

독립적인 파일 여러 개를 조사하거나 수정할 때 subagent를 병렬로 사용할 수 있습니다.

Use subagents to find and update all templates affected by this change. Each subagent should work on an independent template group and report modified files and tests.

⚠ 주의 — 파일 충돌

  • 병렬 수정은 worktree나 명확한 파일 소유권이 있을 때 안전합니다
  • 무작정 병렬화하면 merge conflict와 리뷰 비용이 늘어납니다

여러 agent가 병렬로 일하고 서로 조율해야 한다면 subagents보다 agent teams를 고려하세요.

Part 5 · 3

자동 테스트만으로는 부족하다

자동 테스트가 통과해도 — 서버가 안 켜질 수 있습니다. UI가 깨질 수 있습니다. 콘솔 에러가 있을 수 있습니다.

Agentic Manual Testing

Python library → python -c로 실제 함수 호출

JSON API → dev server 띄우고 curl로 호출

CLI tool → 샘플 input으로 실행

Web UI → Playwright, Chrome extension, screenshot

→ 수동으로 발견한 문제는 다시 자동 테스트로 흡수하세요.

Part 5 · 4

UI는 눈으로 확인시켜라

UI 작업에서 "구현해줘"만 말하면 안 됩니다. UI는 실제로 봐야 합니다.

구현 후 Playwright로 페이지를 열어라. 또는 Claude Code Chrome extension으로 현재 UI를 확인하라. 스크린샷을 찍고 디자인 스크린샷과 비교하라. 차이점을 목록화하고 수정하라. accessibility tree도 확인해 버튼 label과 heading 구조가 올바른지 검증하라.

Claude는 screenshot과 accessibility tree로 꽤 많은 UI 문제를 잡습니다.

최종 판단은 사람이 — 하지만 1차 검증을 해주면 리뷰 비용이 줄어듭니다.
